# Must-buy selection, conviction weighting, champion re-rating, and model improvements

Date: 2026-07-10
Status: approved

## Background

Both models currently force exactly `N_PICKS = 10` names per month. The K-means
model (currently labeled "challenger") has decisively outperformed the LightGBM
model (currently "champion") — CAGR 25.6% vs 7.9%, Sharpe 0.92 vs 0.39 — after
the min-cluster-size bug fix documented in `improvement/improve_v1.md`. That
investigation also established that (a) none of the 14 features carries
meaningful standalone monthly signal, and (b) momentum ranks noticeably better
*within* a cluster of similar stocks than across the whole universe.

This design covers four changes:

1. Variable-count "must-buy" selection replacing fixed top-10.
2. Conviction-weighted portfolio output (percentage allocations).
3. Re-rating: K-means becomes champion via rename + role pointer.
4. An improvement workstream (K-means tuning + new features) with an honest
   validation protocol.

## 1. Must-buy selection

Each model emits only the names clearing a conviction bar; the count varies
month to month.

- **K-means model**: cluster the universe into `K_CLUSTERS` groups on the
  standardized features (as today), identify the momentum cluster by highest
  mean composite momentum, then must-buys = cluster members whose composite
  momentum z-score (vs the whole universe) is **> MUST_BUY_Z_KMEANS (default
  0.0)**, i.e. above universe average.
- **LightGBM model**: score all stocks with the walk-forward model, standardize
  scores cross-sectionally within the month, must-buys = names with
  **z ≥ MUST_BUY_Z_LGBM (default 1.0)**. A z-threshold is used instead of a raw
  predicted-percentile bar because regression predictions compress toward the
  center when signal is weak; a raw 0.9 bar would select ~0 names most months.

### Guardrails (both models)

- **Floor**: fewer than `MIN_PICKS = 5` must-buys → emit nothing for that
  month. The backtest engine and live holdings already interpret a missing
  month as *hold the previous portfolio* (same mechanism as the existing
  min-cluster-size fix).
- **Cap**: more than `MAX_PICKS = 10` must-buys → keep the 10 highest-signal
  names.

New config values: `MIN_PICKS`, `MAX_PICKS`, `MUST_BUY_Z_KMEANS`,
`MUST_BUY_Z_LGBM`. `N_PICKS` is removed once nothing references it.

## 2. Conviction weighting

New helper in `portfolio/weights.py`:

```python
def conviction_weights(signals: dict[str, float], cap: float = 0.20) -> dict[str, float]
```

- Weight proportional to signal strength — the same z-scores used for
  selection, which are strictly positive by construction since both must-buy
  thresholds are ≥ 0 — renormalized to sum to 1.
- Per-stock cap of `WEIGHT_CAP = 0.20`, excess redistributed proportionally to
  uncapped names (water-filling) until no name exceeds the cap. At the floor
  of 5 names this converges to equal 20% weights, so the portfolio is always
  fully invested (5 × 20% = 100%).
- Replaces `equal_weights` for both models' backtest picks and live holdings.
- `equal_weights` remains for fallbacks and the bootstrap.
- The max-Sharpe `_ms` variants are unchanged in method; they now run over the
  must-buy set instead of the fixed top-10.

### Bootstrap fairness

`random_portfolio_bootstrap` currently draws exactly 10 names per month. It
will instead draw the same number of names the strategy actually held that
month, so the null distribution stays comparable under variable counts.

## 3. Rename + champion pointer

Names describe what the models *are*; the champion designation is a pointer.

- Files: `models/challenger.py` → `models/kmeans.py`,
  `models/champion.py` → `models/lgbm.py`. Public functions renamed
  accordingly (e.g. `kmeans_top10` → `kmeans_must_buys`,
  `champion_picks` → `lgbm_picks`).
- Series keys everywhere: `kmeans`, `lgbm`, `kmeans_ms`, `lgbm_ms`, `spy`.
- `config.CHAMPION = "kmeans"` is the single source of truth. Dashboard shows
  a "Champion" badge on that model and orders it first. Future re-ratings are
  a one-line change.
- Registry entries for newly trained LightGBM models use `type: "lgbm"`.

### One-time migration script

`scripts/migrate_model_names.py`, idempotent:

- `models/registry.json`: `type: "champion"` → `"lgbm"`.
- `data/outputs/holdings_champion.json` → `holdings_lgbm.json`,
  `holdings_challenger.json` → `holdings_kmeans.json`; same for
  `leaderboard_*.json`.
- `data/outputs/live_track.parquet`: remap the `model` column values.
- Backtest output files are regenerated by the report rerun, not migrated.

## 4. Improvement workstream

### Validation protocol (applies to all tuning and feature work)

- **Development period**: history start → 2019-12.
- **Holdout**: 2020-01 → present.
- All tuning and feature selection uses only the dev period. The winning
  configuration gets exactly **one** evaluation on the holdout and is adopted
  only if it does not degrade there. No iterating against the holdout.

### 4a. K-means tuning — `scripts/tune_kmeans.py`

Grid:

- k ∈ {3, 4, 5, 6}
- momentum composite ∈ {mean(3/6/12m) (current), 12-1 skip-month momentum,
  volatility-scaled momentum}
- cluster-selection rule ∈ {mean composite (current), risk-adjusted
  (mean ÷ within-cluster dispersion)}
- must-buy threshold z ∈ {0, 0.25, 0.5}

Each config is scored by portfolio Sharpe on the dev period (full pipeline:
selection → conviction weights → backtest engine with costs). The script emits
a ranked report; the winner is confirmed once on the holdout; adopted values
land in `config.py`.

### 4b. New features — `features/`

Evaluated by the same protocol; promoted into `config.FEATURES` only if they
improve LightGBM's dev-period walk-forward IC.

- **Cluster-relative momentum**: per month, z-score each momentum feature
  against the stock's K-means cluster peers (not the whole universe). The
  feature pipeline gains a clustering step so both models can consume these
  columns. Directly operationalizes the within-cluster finding in
  `improvement/improve_v1.md`.
- **Residual momentum**: 12-month cumulative return on the residual after
  removing FF5 factor exposure (using the rolling betas already computed),
  scaled by residual volatility (Blitz/Huij/Martens formulation). No new data
  source required.
- **Vol-scaled momentum**: `return_12m ÷ realized vol`.
- Short-term reversal is already represented by `return_1m`; no new column.

### Explicitly out of scope (and why)

- LightGBM hyperparameter search / alternative boosters / neural nets: the
  investigation showed the features, not the algorithm, are the bottleneck
  (a linear model matched LightGBM's poor result).
- Model ensembling: dilutes the K-means signal while LightGBM is this weak.
- 3-month horizon variant: deferred; changes the cadence of the whole
  pipeline.
- Fundamentals-based features: separate, larger project.

## 5. Surfaces, tests, cleanup

- **Dashboard** (`app/streamlit_app.py`): per-model "portfolio manager" view —
  table of current must-buys with % allocations, pick count for the month,
  champion badge from `config.CHAMPION`. Metrics/equity curves re-keyed to the
  new series names.
- **Tests** (TDD): update `test_challenger` (→ `test_kmeans`), `test_daily`,
  `test_backtest_report`; new tests for `conviction_weights` (proportionality,
  cap + redistribution, sums to 1), both must-buy rules, the 5/10 guardrails,
  and migration idempotency.
- **Repo hygiene**: verified no action needed — the `* 2` duplicate artifacts
  exist only inside `.venv/site-packages` (pip internals, gitignored), not in
  the repo.
- **Backtest refresh**: rerun the full report; commit new
  `metrics.json` / `picks.json` / curves as the new baseline.

## Acceptance criteria

1. Monthly pick counts vary and always lie in [5, 10] (or the month is a
   hold); no month is ever 100% one stock.
2. Every emitted portfolio's weights sum to 1 with no weight above 20%.
3. Dashboard shows K-means as champion with a percentage-allocation portfolio
   table for each model.
4. Registry, holdings, leaderboard, and live-track history read consistently
   under the new names after migration.
5. Tuning/feature adoption decisions are documented with dev-period and
   holdout numbers in the tuning report output.
6. Full test suite passes; backtest report regenerates end to end.
